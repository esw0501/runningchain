1이 프로그램이 하는 일

러닝체인은 "폰 2대 · 거래소 2곳"을 이용한 무위험 차익거래를 설계·계산하고, 결과를 가계부처럼 기록하는 웹 도구입니다.

🎯 전략 핵심 원리

거래소가 신규 가입자·이벤트 참여자에게 주는 쿠폰·보너스를 "공짜 버퍼"로 사용해서, 가격이 어느 방향으로 움직이든 이론상 이익을 내는 구조입니다.

작동 구조

왜 이익이 보장되나?

  1. 가격이 상승하면 A(LONG)가 이익, B(SHORT)가 손실 → 손실은 쿠폰이 먼저 소진되므로 실제 원금 피해 없음
  2. 가격이 하락하면 반대로 B가 이익, A가 손실 → 역시 A의 쿠폰이 먼저 소진
  3. 결국 양쪽에 충전된 쿠폰 합계 − 수수료·펀딩비 = 순수익
중요
"무위험"은 이론적입니다. 실제로는 ① 두 거래소 가격차(베이시스), ② 펀딩비, ③ 거래소 약관 변경, ④ 청산 슬리피지 등 변수로 손실 가능. 러닝체인은 이런 비용을 미리 계산에 반영해서 "그래도 플러스인가?"를 보여주는 도구입니다.

2기본 개념 총정리

📘 꼭 알아야 할 용어

용어쉬운 설명
LONG 롱"가격이 오를 것" 베팅. 오르면 이익.
SHORT 숏"가격이 내릴 것" 베팅. 내리면 이익.
HEDGE 헤지LONG과 SHORT를 동시에 진입해서 가격변동 자체를 중화.
레버리지내 돈의 몇 배를 베팅하는지. 10× = 1만원으로 10만원어치 포지션.
원금 (Margin)내가 실제로 낸 돈. 손실은 여기서 차감.
쿠폰/보너스거래소가 이벤트로 지급. 손실 방어용으로 원금보다 먼저 소진되거나, 원금과 함께 차감되거나, 청산 시 회수됨 (거래소마다 다름).
청산 (Liquidation)손실이 원금+쿠폰을 초과할 듯하면 거래소가 강제로 포지션을 닫음. 원금 대부분이 날아감.
노셔널 (Notional)포지션의 총 크기. 원금 × 레버리지. 수수료·펀딩비는 여기에 부과.
베이시스 (Basis)같은 BTC라도 거래소마다 1~5 USDT씩 다름. 헤지 불완전성 비용.
펀딩비 (Funding)8시간마다 롱·숏이 서로 주고받는 이자 비슷한 비용. 양수면 롱이 숏에게 지불.
TP / SLTake Profit(익절) / Stop Loss(손절). 자동 청산 가격 설정.
BE (Break-even)손익 0 지점. 수수료까지 포함하면 진입가보다 살짝 더 높거나 낮음.

🧮 쿠폰 손실 방식 3가지 (중요!)

거래소마다 쿠폰이 어떻게 소진되는지 다릅니다. 이걸 잘못 설정하면 계산이 완전히 틀어집니다.

모드대표 거래소동작
원금 우선 (principal-first)Bybit손실 → 먼저 원금 깎임 → 원금 다 쓰면 쿠폰 깎임
동시 차감 (simultaneous)AIVA손실 → 원금과 쿠폰 비율대로 동시 차감
청산 시 회수 (voucher-recall)Bitget청산되면 쿠폰은 회수(=0), 원금만 남음
경고
실제로 거래소에 문의 또는 소액(50 USDT)으로 테스트해서 쿠폰 규칙을 반드시 확인하세요. 약관은 자주 바뀝니다.

3Simple 모드 3단계 사용법

처음 써본다면 Simple 모드부터 시작하세요. 입력은 3단계만 거치면 됩니다.

1
전략 선택

"HEDGE"는 양폰 동시 진입(무위험 차익거래). "LONG ONLY"·"SHORT ONLY"는 한쪽만 단방향. 처음이면 무조건 HEDGE.

2
거래소 선택

A거래소·B거래소를 각각 고릅니다. 일반적으로 A = 기존에 쓰던 익숙한 거래소, B = 쿠폰 주는 이벤트 거래소. 쿠폰 손실 방식이 자동 설정됩니다.

3
원금·레버리지·쿠폰 입력

슬라이더를 움직이면 결과가 실시간으로 아래 박스에 표시됩니다. 원금은 보통 양쪽 같게, 레버리지는 10× 권장(초보). 쿠폰은 해당 거래소가 실제로 지급한 금액.

팁
결과 박스의 "상승시 / 하락시" 두 숫자가 모두 양수이면 무위험 헤지가 성공한 것. 한쪽이 음수로 나오면 비용이 쿠폰보다 커서 손해 → 원금이나 쿠폰을 조정해야 합니다.

4Expert 모드 심화

Simple 모드의 기본값(기준가 76,048 · 변동률 ±2.5% · 8시간 보유)을 바꾸고 싶거나, 세밀한 청산가·비용 분석이 필요할 때 Expert 모드를 씁니다.

주요 추가 설정

  • 베이시스 (Basis Spread): 두 거래소 가격차. BTC는 보통 1~5 USDT.
  • 보유 시간·펀딩비 A/B: 장기 보유 시 펀딩비 누적. A·B 거래소 펀딩율이 다르면 차액이 비용으로 쌓임.
  • 안전 버퍼: 손절가(SL)가 청산가(LIQ)에 너무 가까우면 실제 청산으로 미끄러질 수 있음. 버퍼 이상이어야 안전.
    • 초록: 여유 충분 (버퍼의 1.5배 이상)
    • 노랑: 경계 (버퍼 1~1.5배)
    • 빨강: 위험 (버퍼 미만)
  • 포지션 자동 일치: HEDGE 모드에서 A·B 노셔널(원금×레버리지)을 강제로 같게 맞춤. 레버리지가 다를 때 체크하면 자동 동기화.

🎨 Risk-Reward 타임라인 읽는 법

포지션 카드 하단의 가로 바는 청산·손절·본전·진입·익절의 가격 거리를 시각화합니다.

왼쪽에 가까울수록 위험(LIQ/SL), 오른쪽이 목표(TP). SHORT 포지션은 반대로 읽습니다 (왼=TP/이익, 오른=LIQ/위험).

5Advanced Tools

Expert 모드 하단의 "🔧 Advanced Tools"를 펼치면 5개 탭이 나옵니다.

① 역산 TP/SL

"400 USDT 벌고 싶다 → 그러려면 TP 얼마?"를 역으로 계산. 목표 순수익 입력 → 필요 가격 상승률·TP 가격 자동 출력.

② 물타기/불타기

포지션이 손실 중일 때 추가 진입으로 평단을 조정. 대상 폰(A/B)·추가 원금·추가 진입가 입력 → 신규 평균단가·총 포지션 확인 → "양식에 반영" 클릭하면 입력란에 자동 반영.

③ 임의 청산

"지금 청산하면 얼마?"를 실시간 체크. A·B 각각의 현재가 입력 → 쿠폰 손실 배분까지 반영한 순손익 출력.

④ A 생존 시나리오 NEW

이 프로그램의 핵심 전략입니다. B거래소(쿠폰 많이 주는 이벤트 거래소)가 청산당했다고 가정하고, A거래소에서 물타기로 얼마나 버텨야 전체 본전인지 역산합니다.

┌─ A 생존 시나리오 ────────────────────────────┐ │ B 가정 청산가: [ 78,000 ] │ │ A 추가 물타기: [ 1,000 ] USDT │ │ [ SIMULATE ▶ ] │ ├───────────────────────────────────────────────┤ │ B 청산 @ 78,000 · 순손실 −2,150 USDT │ │ │ │ A 조정 결과 (물타기 1,000 USDT): │ │ • 신규 평단: 75,032 (기존 76,048) │ │ • 총 포지션: 3,000 · 노셔널 30,000 │ │ • 본전까지 필요 상승: +7.17% → 80,412 │ │ │ │ ⚠ 회복 가능하나 시간 소요 │ └───────────────────────────────────────────────┘
사용 팁
진입 전에 미리 "최악 시나리오(B 청산)"를 돌려보고, 내가 A쪽에 얼마를 더 투입할 수 있는지 한계를 정해두세요. 색상이 초록(<5%)이면 안전, 노랑(5~15%)은 버틸 만, 빨강(>15%)은 추가 물타기 필요.

⑤ 승률 추정 NEW

쿠폰이 방어해주는 가격 범위와, 그 밖으로 벗어날 확률을 변동성 기반으로 추정.

  • 일 변동성 σ: BTC 기준 최근 30일 평균 2~3%. 장세 변동 크면 4~5%.
  • 보유 기간: 포지션 유지 예상 일수.
  • 출력: 방어범위 내 유지 확률 %, 이탈 확률, 안전도 판정.
주의
이 계산은 기하브라운 근사로, 극단 급변동(fat-tail)은 실제보다 낮게 표시됩니다. 참고용으로만 사용하고 실전에서는 더 보수적으로 판단하세요.

6Journal (가계부)

계산만 하고 끝이 아니라, 실제 거래 결과를 기록해야 전략이 통계적으로 유효한지 알 수 있습니다.

🗂 작업 흐름

  1. 계산기에서 시뮬레이션 → 📌 계획 저장 클릭 → Journal에 planned(계획) 상태로 기록
  2. 실제 거래 체결 → Journal에서 해당 행의 진행 버튼 → open(진행중)
  3. 청산 후 → 완료 버튼 → 실현값 기록 모달에서 A·B 실제 청산가, 메모 입력
  4. 대시보드에서 일·주·월 누적 수익 확인, 히트맵에서 거래 빈도·성과 확인

📤 내보내기

  • CSV 내보내기: 엑셀/구글시트에서 추가 분석 가능. ID·일시·전략·원금·쿠폰·계획/실현·편차·ROI·거래소·메모 모두 포함.
  • 전체 삭제: 테스트 데이터 정리용. 실거래 기록 시 주의!

🔍 필터

상태별(계획/진행중/완료/취소), 전략별(HEDGE/LONG/SHORT) 필터링. 대시보드에서는 기간별(7D/30D/90D/YTD/ALL) 칩으로 필터.

7진입 전 체크리스트

실제 거래 전에 아래 항목을 하나씩 확인하세요. 하나라도 빠지면 "무위험"이 아닙니다.

✅ 거래소

  • A·B 모두 같은 종목인가? (BTCUSDT Perpetual)
  • 쿠폰 손실 방식을 실제로 확인했는가?
  • 출금·거래 제한은 없는가?
  • KYC·2FA 완료?

✅ 포지션

  • A·B 노셔널이 동일한가? (자동 일치 체크)
  • 레버리지 동일한가?
  • 청산가와 SL 사이 안전 버퍼 확보?
  • 기준가 차이(베이시스) 반영?

✅ 비용

  • 수수료율 맞게 설정?
  • 예상 보유 시간 펀딩비 차감 후에도 플러스?
  • 두 거래소 펀딩 방향이 같은가?

✅ 대비

  • A 생존 시나리오 돌려봤는가?
  • 최악의 경우 추가 투입할 금액 결정?
  • Journal에 계획 저장했는가?
절대 금물
  • 레버리지 50× 이상: 작은 변동에도 청산. 초보는 10× 이내.
  • 쿠폰 규칙 미확인 진입: 쿠폰이 원금과 함께 날아가면 계산이 무너짐.
  • 한쪽만 먼저 체결: 베이시스 순간 스파이크로 진입가 차이 벌어짐 → 양쪽 동시 체결 필수.
  • 월 허용 손실 초과: 설정에서 월 리스크 한도 정하고 넘으면 중단.

8자주 묻는 질문

Q1. 데이터가 어디 저장되나요? 서버에 올라가나요?
A. 브라우저 localStorage에만 저장됩니다. 서버 전송 없음. 브라우저 데이터를 지우거나 다른 기기에서 열면 초기화. 중요한 기록은 CSV 내보내기로 백업하세요.
Q2. 모바일에서도 쓸 수 있나요?
A. 560px 이하 모바일 레이아웃 지원. 하지만 양폰 거래 자체는 PC에서 계획, 폰 2대에서 실행이 권장 흐름.
Q3. 계산 결과가 실제 거래와 왜 달라요?
A. 주요 원인: ① 슬리피지(체결가 지연) ② 거래소 약관 변경 ③ 펀딩율 실시간 변동 ④ 쿠폰 소진 공식이 실제로 다를 경우. 러닝체인은 "이론값"이고, 실측은 Journal에 기록해서 편차를 확인하세요.
Q4. "포지션 자동 일치"는 언제 써요?
A. A·B 거래소의 레버리지가 서로 다를 때. 예: A 20× / B 10× 라면 같은 원금으로 포지션 크기가 2배 차이. 체크하면 B 원금을 자동으로 2배 맞춰 노셔널 동일화.
Q5. 승률 추정이 85%인데 실패했어요. 왜요?
A. 승률은 정규분포 가정으로 추정. 암호화폐는 실제로는 꼬리(tail)가 두꺼워 극단 변동이 더 자주 나옵니다. 85%는 "장기적으로 그럴 것"이지 개별 거래 보장이 아닙니다. 쿠폰 방어 범위가 넓을수록 안전.
Q6. 쿠폰이 "동시 차감(AIVA형)"인데 원금 우선으로 잘못 설정하면?
A. 실제 손실이 계산보다 더 크게 나올 수 있습니다. 원금 우선은 쿠폰이 나중에 쓰여서 원금 보호가 강함. 동시 차감은 원금도 즉시 깎임. 반드시 거래소에 확인하세요.
Q7. 베이시스·펀딩비는 어디서 확인해요?
A. 각 거래소 종목 페이지에 "Mark Price"와 "Funding Rate"가 표시됩니다. 진입 직전에 확인하고 입력값을 맞추세요.
